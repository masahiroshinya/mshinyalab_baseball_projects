# GitHubとデータを同期する基本の5ステップ

新しく作業を始める時から、GitHubへ反映させるまでの「基本の5ステップ」です。
複数人で同じプロジェクト（GitHub）を触る場合、**「作業前に最新状態にする」**ことと**「作業後に保存して送る」**ことがセットになります。

## Step 1: 作業を始める前に、最新のデータを取り込む（Pull）
作業を開始する前に、共同管理者が何か新しいデータをGitHubにアップしていないか確認し、自分のPCに取り込みます。

```bash
git pull origin main --rebase
```

> **Tip!**
> これを忘れて古いデータを編集してしまうと、あとから合流させるのが大変になります。「作業開始時はまず pull」と覚えておくと安心です。

## Step 2: いつも通りローカルPCで作業する
VS Codeなどを使って、ファイルを作成したり、記事を書いたり、コードを修正します。この段階ではまだGitには記録されず、普通にPCに自動保存されているだけです。

## Step 3: 変更したファイルをリストアップする（Add）
作業が終わったら、GitHubに送りたいファイルの変更を「送信リスト（ステージングエリア）」に乗せます。
以下のコマンドで、変更されたすべてのファイルをまるごとリストに乗せることができます。

```bash
git add .
```
*(※最後の `.` （ドット）は「ここにある変更全部」という意味の記号です)*

## Step 4: 変更をセーブ（確定）する（Commit）
リストに乗せた変更を、あなたのPC内の記録として確定（セーブ）します。このとき「何を変更したか」のわかりやすいメモを残します。

```bash
git commit -m "今日読んだ〇〇の先行研究のメモを追加"
```
*(※ `-m` の後のダブルクォーテーション `"` で囲まれた部分に、日本語で自由に変更内容を書きます)*

### なぜわかりやすいメモ（コミットメッセージ）が必要なのか？
1. **チームメンバー（共同管理者）に作業内容を伝えるため**
   中身を一行ずつ読まなくても「なぜ、何の目的で書き換えたのか」が一目でわかるようにするため。
2. **過去の自分のため（強力な「タイムマシン機能」）**
   何かトラブルがあって過去のデータに戻りたいとき、「この時に何をしたか」が明確であればすぐにお目当てのセーブデータに戻ることができます。
3. **バグやトラブルの原因を特定しやすくするため**
   エラーが起きた際、履歴を遡って原因を特定しやすくなります。

**【良いコミットメッセージの例】**
- `[追加] Kida et al. 2005 の文献ノートを作成`
- `[修正] 文献整理ルールに概要セクションの書き方を追記`

## Step 5: GitHubへ送信する（Push）
ここまできたらあとはネット上に送信するだけです！

```bash
git push origin main
```

---

## 補足：もしStep 5でエラー（[rejected]）が出たら？
あなたが作業をしている間に、共同管理者の先生が先にGitHubへ新しいデータを反映させていた場合、Step 5で安全装置が働きブロックされます（エラーになります）。

その場合は慌てずに、**もう一度ネット側の新しいデータを取り込んでから、再送信**すればOKです（ローカルのデータが勝手に消えることはありません）。

```bash
git pull origin main --rebase
git push origin main
```
